# Prompt Design Philosophy V1.0
（从 0 到 1 的“认知激发”指南）

> **定位**：这是一份“Prompt 设计哲学”文档——关注如何用 Prompt 激发模型能力、塑造生成分布、建立可对齐的思维场域。  
> **不做的事**：不把工程规范、格式校验、工具调用、修复回路等“可控性工程”写进来（这些留在 **PromptEngineeringV3**）。  
> **你可以把它理解为**：先学“导演与美学”，再去看“施工图与验收标准”。

---

## 0. 一句话总纲：Prompt 不是指令，是“场域”
把 Prompt 当成“给机器下命令”时，你自然会写出越来越细的规则；  
把 Prompt 当成“搭台与布光”时，你会开始关心：  
- **注意力会被吸到哪里**（信息权重与焦点）  
- **思考姿态是什么**（角色、方法、风格）  
- **如何自我收敛**（评判系统与迭代）  

因此，一个更好用的定义是：

> **Prompt = 场域设计（Context Field Design）**  
> 你不是在写约束，而是在写“引力”。  
> 你不是在编程模型，而是在让语义产生共振。

---

## 1. Prompt 的三件本质工作
从 0 到 1 的 Prompt，通常只需要回答三类问题：

1. **你要优化的到底是什么？**（意图 / 目标函数）
2. **模型“相信自己身处何处”？**（世界 / 场景 / 受众）
3. **它用什么方式思考并自我对齐？**（方法 / 裁判）

把这三件事做对，很多“规范”无需写，模型会自动补齐合理行为；  
反过来，如果这三件事缺失，再多规则也像“在真空里写施工规范”。

---

## 2. 四元组骨架：意图 / 世界 / 方法 / 裁判
这是最适合从 0 开始创作 Prompt 的最小骨架。  
每次写 Prompt，先把四个槽位填满，再决定要不要工程化。

### 2.1 意图 Intention：你到底在优化什么
从 0 到 1 的失败常见原因不是模型不行，而是你没说清“最重要的是什么”。

用三个问题锁定意图：

- **任务形态**：探索（发散）还是决策（收敛）？  
- **成功标准优先级**：新颖性 / 正确性 / 可执行性 / 说服力 / 风格一致性，谁第一？  
- **边界条件**：允许虚构吗？允许推断吗？证据不足怎么办？

> 哲学提示：意图不是“输出长什么样”，而是“你希望在读者脑中发生什么改变”。

---

### 2.2 世界 World：让模型“进入某个工作模式”
世界不是背景装饰，而是模型行为策略的开关。世界搭得好，模型会自动选择更贴合的表达与结构。

构建世界时，优先给四类信息：

- **身份与伦理**：它是谁？负责什么？不负责什么？  
- **场景与受众**：写给谁看？用于什么场合？（内部备忘录 / 客户提案 / 课堂讲解 / 深夜电台）  
- **资源与素材**：可用材料在哪里？哪些只能参考不能执行？  
- **风险与 stakes**：错了会怎样？要不要保守、要不要多方案、要不要标注不确定？

#### 2.2.1 斯坦尼斯拉夫斯基法则：别只给“职称”，给“生命体验”
- “你是资深程序员”只是标签  
- “你写代码像工匠打磨家具，崇尚简洁与可读性”会激活价值观与审美，让输出更一致

> 你在给模型的是一组“人格向量”，而不只是一个角色名。

#### 2.2.2 认知锚点：用高密度概念瞬间对齐思维框架
在 Prompt 中点名某个思维模型/流派，往往比写一页说明更有效，例如：
- 第一性原理（逼它追本溯源）
- 苏格拉底式反问（逼它用提问引导）
- 费曼技巧（逼它通俗解释）
- 系统论、博弈论、叙事学、产品思维、编辑思维……

> 这些词是“语义压缩包”，会触发整套隐含框架。

---

### 2.3 方法 Method：你希望它以什么“认知姿势”思考
方法不是步骤列表，而是“让模型切换姿态”的指令：发散、对比、批判、模拟、重写、抽象、落地……

常用的三种方法姿势（从 0 到 1 最稳）：

1. **先发散后收敛**：先要多样性，再做取舍  
2. **先定裁判再生成**：先写 rubric/评估维度，再产出  
3. **先拟合风格再填充内容**：先确定语气与结构，再写内容

#### 2.3.1 思维路径仿真：让“好答案”成为“好推理”的副产品
你关注的不是“它算没算对”，而是“它沿着什么路径走到这里”。

你可以用两种非工程化方式激发高质量推理：

- **草稿阶段**：先列 3–6 个方向 + 取舍理由，再生成最终稿  
- **反例测试**：让它指出方案在哪些情况下会失败，并给出补救

#### 2.3.2 焦距控制：升维与降维（Zoom Out / Zoom In）
- 当输出泛泛：要求“具体到动作、句子、示例、数字”  
- 当输出陷入细节：要求“系统论视角 / 历史视角 / 目标函数视角”  

> Prompt 可以像镜头一样拉远拉近；焦距变化本身就是能力召唤。

#### 2.3.3 多声部思考：用“内部合议”获得更稳的洞见
如果你要的是“更强的思考”，而不是“更快的回答”，可以让它模拟：
- 乐观者 / 怀疑者 / 实干家 / 审美编辑  
- 用户视角 / 对手视角 / 投资人视角 / 合规视角

你不是在追求戏剧效果，而是在做“视角覆盖率”的提升。

---

### 2.4 裁判 Judge：用“评判系统”替代“束缚”
从 0 到 1 的质量跃迁，往往来自**裁判系统**，而不是更多规则。

最小可用裁判有两种形态：

- **Checklist**（5–10 条）：覆盖、清晰、风险、风格、可执行性  
- **Rubric**（4–6 维）：准确性、相关性、结构、说服力、创造性、可落地性

> 核心哲学：**少写禁令，多写评审标准**。  
> 让模型自己对齐标准，比你手动“拧螺丝”更有效。

#### 2.4.1 “诚实协议”：把创造力用在方案上，而不是编事实
当你在高风险事实领域（医疗/法律/财务/实时信息）工作时，哲学层面也要给模型一条底线：
- 不确定就标注不确定  
- 证据不足就说明缺什么  
- 不要用“听起来很确定”的语气补空白  

> 这是“创作伦理”，不是工程规范；它保护你的判断力，也保护模型输出的可信度。

## 3. 三阶段创作流程：发散 → 收敛 → 固化
这是一条把“创造力”与“可复用性”兼容起来的路径。

### 3.1 发散：激发能力（让潜空间展开）
目标：让模型把可能性尽量展开。  
做法：
- 给强世界、强意图张力（受众、冲突、限制、期望效果）  
- 少硬性格式，允许多方案输出（3–8 个方向）  
- 要求每个方向附“亮点 / 风险 / 适用场景”

发散阶段的关键不是“更像规范”，而是“更像灵感会议”。

---

### 3.2 收敛：把好点子变成好作品
目标：在多解中选最优，并补齐结构与细节。  
做法：
- 让它比较 A/B/C 的 trade-off  
- 先出大纲再扩写（或分段生成）  
- 引入裁判：按 rubric 打分并解释扣分点

> 哲学提醒：收敛不是删掉想象力，而是把想象力压成密度更高的表达。

---

### 3.3 固化：把一次成功“封装成可复用”
目标：把这次成功的方法沉淀成模板或工作流。  
做法（只写哲学原则，不写工程细则）：
- 把“关键的好”描述成可复查的标准（比如覆盖要点、结构层次、读者行动）  
- 保留“裁判 + 迭代”的机制，而不是堆更多约束  
- 若要进入生产/团队复用，再交给 PromptEngineeringV3 做工程化

---

## 4. 张力管理：自由度 vs 可控性
每个 Prompt 都在管理一条张力轴：

- **开放性**越高：越新颖，但越不稳定  
- **约束性**越高：越稳定，但越容易僵化

哲学解法是“分阶段调参”：
- 发散阶段：少约束、多裁判（先让它自己评）  
- 收敛阶段：增加结构与取舍（大纲、对比、反例）  
- 固化阶段：才考虑把结构变成工程可控的约束

> 你不是在二选一，而是在“按阶段切换模式”。

---

## 5. 注意力经济学：写给“权重系统”看的 Prompt
模型的注意力是一种稀缺资源。你写的每一句话都在争夺权重。

哲学上，你只需记住三条：

1. **把最重要的放在高权重区**（开头/结尾）  
2. **避免把关键内容埋在“中段死区”**（长上下文中部容易被稀释）  
3. **让“指令”和“材料”在语义上分开**（否则会互相污染、互相稀释）

这三条足以解释多数“为什么它没按我说的做”。

---

## 6. 语言即咒语：语义共振与语言密度
Prompt 的词汇密度越高，模型输出越“尖锐”。

### 6.1 强动词 + 具体名词
- “改一下”太弱  
- “打磨成更有节奏、更有冲击力的短句”会给出审美方向  
- “分析”很泛  
- “解剖 / 归因 / 合成 / 批判 / 复写”更能定位思维动作

> 模糊词给模型宽分布；精准词给模型尖分布。

### 6.2 氛围调色板：风格不是装饰，是信息载体
你可以直接写：
- 语气：冷静、克制、锋利、温暖、幽默、学术……  
- 叙事：冰山理论、深夜电台、律师陈词、产品发布会、编辑部改稿……

风格不是“最后加的滤镜”，而是“决定内容如何被组织”的框架。

### 6.3 用对比描述风格：正例 + 反例
如果你很在意某种风格一致性，哲学上最有效的是：
- 给一个你喜欢的“相似样本”  
- 再给一个你讨厌的“反例”  
- 让模型主动总结差异，并以此生成

> 你在训练它识别“美学边界”，而不是列规则。

---

## 7. 共生迭代：把 Prompt 当成活体
Prompt 最强的形态不是“一次写对”，而是“在对话中长出来”。

### 7.1 像指导人一样给反馈
不要只说“太长/不对”。你可以说：
- “结构很好，但缺少同理心；保留结构，语气更温暖。”  
- “这个比喻牵强，换一个更日常的。”  
- “把抽象句子换成 2 个具体场景例子。”

模型对“人类式反馈”反应很强，因为它天然擅长模仿与对齐。

### 7.2 Meta-Prompting：让模型帮你写 Prompt
当你不知道怎么描述需求时，直接让模型来问你：
- “为了更好完成 X，请向我提 8 个澄清问题。”  
- “请把我的需求重写成一个更强的系统提示词草案。”

> 这是一种人机共创：把主导权暂时移交给更擅长语义组织的一方。

---

## 8. 从哲学到工程：两份文档如何分工
你可以把工作流拆成两步：

1. **先用哲学文档写“场域”**：意图 / 世界 / 方法 / 裁判  
2. **再用工程文档做“固化与验收”**：格式、约束、工具、修复、评估、版本管理

一个实用的判断标准：

- 当你在“找方向、找表达、找灵魂” → 用本哲学文档  
- 当你在“要稳定复现、要团队协作、要线上可靠” → 交给 PromptEngineeringV3

## 附录 A：三个通用 Prompt 骨架（场域胚子）
> 下面不是工程模板，而是“场域骨架”。把四元组填进去即可。

### A1. 探索型（发散）
- **意图**：我要 6 个不同方向，优先新颖性与启发性  
- **世界**：你是 ___；受众是 ___；场景是 ___  
- **方法**：先列方向→每个方向亮点/风险/适用场景→给 1 个推荐组合  
- **裁判**：新颖性 × 可执行性 × 契合度（请自评并解释）

---

### A2. 收敛型（决策/成稿）
- **意图**：我要一个可交付版本，优先可执行性与说服力  
- **世界**：你是 ___；我要拿给 ___；决策风险是 ___  
- **方法**：先给大纲→按段生成→自评→精修  
- **裁判**：结构清晰、关键点覆盖、行动明确、语气一致、风险提示

---

### A3. 批判型（编辑/评审）
- **输入**：这是 v1 草稿 + 成功标准  
- **意图**：用最小修改把它变成 v2  
- **方法**：逐条指出偏离标准→给修改建议→输出 v2  
- **裁判**：只改最影响质量的 20%，避免推倒重来

## 附录 B：哲学检查清单（点击发送前问自己）
1. 我是否说清了“最重要的成功标准”？  
2. 我是否搭了一个足够可信的“世界”（身份、受众、场景、风险）？  
3. 我是否指定了希望它采用的“认知姿势”（发散/对比/批判/落地）？  
4. 我是否提供了“裁判”（checklist 或 rubric）让它自我对齐？  
5. 我使用的词是否足够具体、有力、能引发语义共振？
